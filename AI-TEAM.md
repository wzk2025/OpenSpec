# AI 团队能力清单

> 本文件随框架安装到目标工程根目录，供人类工程师快速了解 AI 团队能做什么、怎么用。

---

## 快速开始

在 Claude Code 中，用自然语言向主会话发指令即可启动协作：

```
我需要实现[功能描述]，请按开发流程执行。
验收标准：
1. ...
2. ...
```

主会话会自动调度架构设计、编码实现、质量评审等角色。

---

## AI 团队成员

| 角色 | 职责 | 你需要做什么 |
|------|------|-------------|
| **PM（你）** | 发出指令、定义验收标准、审批结果 | 用自然语言描述需求 |
| **主会话** | 接收指令、调度各角色、执行质量门禁 | 自动运行，你审批最终结果 |
| **Architect** | 架构设计、技术选型、接口定义、SPEC 生命周期管理 | 自动运行 |
| **Developer** | 编码实现、写单元测试（覆盖率 > 80%） | 自动运行 |
| **Tester** | 集成测试、回归测试、E2E 测试 | 自动运行 |
| **Reviewer** | 代码评审、文档质量检查、UX 评估 | 自动运行 |
| **Debate** | 对抗审计：Defender → Attacker → 主会话裁决（≤13 分钟） | 争议方案时自动触发 |

## 你能做什么

### 标准开发
```
我需要实现用户登录功能，支持 JWT 认证和密码加密。
验收标准：1. 登录/注册接口 2. Token 过期自动刷新 3. 单元测试覆盖率 > 80%
```

### 紧急修复
```
紧急修复：生产环境查询接口超时，需要添加缓存层。
```

### 架构评审
```
评审当前的数据库分表方案是否合理。
```

### 规范驱动开发（SPEC 流程）

SPEC 是本框架最核心的开发模式，确保每个功能从需求到实现有完整文档追溯。**推荐所有非紧急功能使用 SPEC 流程。**

#### 整体流程

```
rd:* 命令（4 步）:   /rd:explore  →  /rd:propose  ─────→  /rd:apply  ─────→  /rd:archive
                                    │                  │                │
质量保障环节:                        │  SPEC 评审        │  测试 + 代码评审   │
                                    │  (spec-review)    │  (test+review)   │
                                    ▼                  ▼                ▼
执行角色:         Architect 梳理   Architect 生成    Developer 实施    Architect 归档
                   需求边界         全部制品          按任务逐个实现       生成归档摘要
```

> **命令与质量保障的关系**：rd:* 是 RD CLI 提供的 4 个操作命令，框架在命令之间插入质量保障环节（SPEC 评审、测试、代码评审），确保每个阶段的产出符合质量标准。

#### 你在每一步做什么

| 阶段 | AI 做什么 | 你需要做什么 |
|------|-----------|-------------|
| ① /rd:explore | Architect 深入思考、梳理需求边界 | 描述清楚你想要什么、不想要什么 |
| ② /rd:propose | Architect 产出提案+设计+任务（一次性） | 审阅全部制品，确认需求和技术方案 |
| **SPEC 评审** | Reviewer 用 spec-review-flow 评审文档质量 | 审阅评审报告，不通过则回到 ② |
| ③ /rd:apply | Developer 按任务逐个实现（TDD） | 等待完成，不干预 |
| **测试+代码评审** | Tester 集成测试 → Reviewer 用 code-review-flow 评审代码 | 审阅测试报告和评审报告 |
| ④ /rd:archive | Architect 归档已完成的变更，生成归档摘要 | 确认归档 |

#### 如何启动

```
进入规范模式，我需要实现多租户数据隔离功能。使用 /rd:explore 开始。
验收标准：
1. 不同租户数据完全隔离
2. 租户切换时性能影响 < 50ms
3. 单元测试覆盖率 > 80%
```

主会话会自动按上述流程逐步推进，每个关键节点会请你审批后再继续。

#### SPEC 文档存放在哪

所有文档在 `.harness/spec/` 目录下，按模块组织，你可以随时查阅：
```
.harness/spec/{模块名}/
├── _meta.md              # 该模块所有 SPEC 的状态总览
├── {功能名}-spec.md      # 正式 spec 文档
├── changes/{功能名}/     # 过程文档（提案、设计、任务、评审）
└── archive/              # 已完成功能的归档
```

## 质量保障机制

AI 团队内置多级质量保障：

- **质量门禁**：每个阶段自动评分，不合格自动返工
- **对抗审计**：争议方案由正反两方 AI 独立辩论，主会话裁决
- **PM 审批**：最终结果由你决定是否通过，你有否决权

### 评分规则

| 得分 | 处理 |
|------|------|
| ≥ 90 | 直接通过 |
| 80-89 | Reviewer 修复非 P0 问题后通过 |
| 75-79 | 触发对抗审计（Debate） |
| < 75 | 直接返工 |
| 存在 P0 | 一票否决，得分归零，直接返工 |

### 评分权重

加权平均：Architect(25%) + Reviewer(40%) + Tester(35%)

## 注意事项

1. **你是 PM**，所有工作由你发起指令，主会话负责调度
2. **你是最终决策者**，主会话汇报结果后需要你审批
3. **Git 提交由主会话执行**，AI 开发角色不能直接提交代码
4. **产品级规则**在 `.claude/reference/CLAUDE.md`，由单独团队维护，不可修改
5. **工作流文件**在 `.claude/workflows/` 下：开发流程、评审流程、热修复流程、ADR 流程
